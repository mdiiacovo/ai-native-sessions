Best-in-Class Copilot Prompting

A one-page field guide for the hackathon · print or keep it open

1 Specify the outcome, not the steps

Say what "done" looks like. Let Copilot choose the how.

Good"Add a search box that filters the queue by title or team, matching the existing dark theme."
Avoid"Create a function, then add an input, then loop over the array…"

2 Give constraints

Stack, style, and what not to touch. Guardrails raise quality.

Good"Keep it zero-dependency (vanilla JS). Don't change the data model or add packages."

3 Iterate in small loops

One change at a time, run it, then refine. Beats one giant mega-prompt.

Good"Now add a risk-level dropdown too, and update the count badge to reflect filters."

4 Ask it to plan first

Get the approach before the code. Catch wrong turns early — cheaply.

Good"Before coding, outline your plan to add filtering without new dependencies."

★ Ground the model

Great context → great output. Beginners skip this; pros never do.

  • Open (or reference) the files that matter
  • Point at the symbol, function, or example to mirror
  • Use copilot-instructions.md to pin project conventions
  • Paste the error, the log, or the spec — don't make it guess

⚑ Anti-patterns to kill

  • Vague one-liners ("make it better")
  • One massive prompt doing ten things
  • Blindly accepting output without reading it
  • No constraints, then surprised by the stack it picked
  • Re-prompting from scratch instead of correcting

⟳ The core loop

Context→ Prompt→ Build→ Delegate→ PR

Small enough to code yourself → do it. Big enough to hand off → assign the issue to the Copilot coding agent and get a pull request back.

🏁 Hackathon quick wins

  • Start from a written issue — clarity compounds
  • Delegate boilerplate; spend your time on the hard 20%
  • Ask Copilot to review for security issues before judges do
  • Make it demo-proof: README, a clear story, measurable impact
